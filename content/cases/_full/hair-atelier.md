# Hair Atelier — сайт-визитка и CRM салона красоты

**Тип:** Сайт салона + админка с управлением записями, мастерами, финансами (конкурент YCLIENTS / DIKIDI)
**Год:** 2026

## О проекте

Полностью самописная платформа для салона красоты: публичный сайт-визитка
с онлайн-записью и полнофункциональная админ-CRM для управления салоном.
Прямой конкурент YCLIENTS и DIKIDI, но без подписки и привязки к чужой
инфраструктуре — весь код, база и Telegram-боты живут на одном VPS клиента.

Поверх обычной «записывалки» — собственная система клиентских сообщений
(prompts), автогенерация повторных записей, отдельный Telegram-бот для
администраторов и мастеров с инлайн-управлением записями, и
персонализированные обращения в дательном падеже через `petrovich`.

## Структура сайта

### Публичная часть (mobile-first)

- **Главная** — герой, услуги, мастера, CTA, блок «следующая запись» в
  личном кабинете, стек активных сообщений от салона
- **/services** — каталог услуг с категориями, ценами от/до, длительностью
- **/booking** — пошаговая форма записи (мастер → услуга → дата/время →
  контакты) с предварительным выбором мастера и услуги через query-параметры
- **/shop** — витрина бьюти-товаров
- **/school** — курсы салона с записью
- **Личный кабинет клиента** — JWT по телефону, список своих записей,
  ответы на сообщения салона

Адаптив: **нижний tab-bar на мобильных, верхний хедер на десктопе**,
кастомный pull-to-refresh на iOS/Android.

### Админ-панель (/admin)

- **Дашборд** — записи сегодня, ожидающие подтверждения, на неделе,
  всего; последние записи; быстрые ссылки на разделы
- **Календарь** — недельная сетка по мастерам, 30-минутные слоты,
  цветовые статусы (PENDING / CONFIRMED / COMPLETED / CANCELLED),
  модалка записи с оплатой и сменой статуса, блок-слоты, схема
  расписания мастера на день
- **Записи** — CRM-таблица со всеми записями, фильтры, статусы,
  пагинация
- **Клиенты** — клиентская база с историей визитов, заметками мастера,
  предпочтениями (аллергии, пожелания по окрашиванию), детальная карточка
- **Финансы** — отчёт по выручке за период (пресеты «сегодня / 7 / 30 /
  90 дней» или произвольные даты), разбивка по способам оплаты
  (наличные / карта / онлайн), по мастерам и по услугам
- **Сообщения** — система клиентских промптов (см. ниже): входящие
  ответы с фильтром по статусу, конструктор сообщения с вариантами
  ответа, библиотека шаблонов
- **Мастера** — CRUD, привязка к услугам с индивидуальной ценой,
  расписание по датам, блок-слоты, привязка к Telegram-аккаунту
- **Услуги / категории / товары / курсы** — CRUD с сортировкой
- **Запись на курсы** — отдельная таблица с уведомлениями в Telegram

## Технические решения

### Календарь записей

Главная рабочая поверхность администратора. Не таблица, а полноценная
графическая сетка времени.

- Колонки — мастера активные на выбранную дату, строки — 30-минутные
  слоты с 09:00 до 21:00
- Запись отрисовывается блоком высотой пропорциональной длительности
  услуги (`durationMin / 30 * 36px`), с цветной полоской статуса слева
- Заблокированные слоты (`BlockedSlot`) — серая заливка с комментарием
- Клик по записи открывает модалку: имя, телефон, услуга, цена мастера
  (если задана индивидуальная, иначе базовая), кнопки смены статуса
  и инлайн-форма приёма оплаты с выбором метода
- Навигация по неделям, переход «сегодня», быстрая ссылка на список
  записей за день

### Индивидуальные цены мастера

Junction-таблица `MasterService` хранит не просто связь, а опциональные
поля `priceFrom` / `priceTo` — каждый мастер может иметь свою цену
за услугу (топ-мастер дороже джуниора). При расчёте цены и оплаты
используется правило fallback: «цена мастера, иначе базовая цена услуги».
Изменение базовой цены не ломает цены топ-мастеров.

### Расписания и блокировки

- `MasterSchedule` — рабочие часы мастера на конкретную дату
  (по умолчанию 09:00–19:00, кастомизируется)
- `BlockedSlot` — перерывы, отлучки, личные дела — точечно по
  30-минутным слотам с комментарием
- Публичная форма записи учитывает и расписание, и блок-слоты, и
  занятые ячейки — клиент физически не может выбрать конфликтное время

### Система клиентских сообщений (prompts)

Собственная замена SMS-маркетингу YCLIENTS. Каждый промпт — это
вопрос клиенту с заранее заданными вариантами ответа, отображается
ему в стеке на главной личного кабинета.

- **SYSTEM-промпты** — автогенерируются после визита. Через
  `REPEAT_BOOKING_DELAY_DAYS` после последней записи клиенту приходит
  «Повторить запись к мастеру Анне на «Окрашивание»?» с кнопками «Да /
  Нет, спасибо». «Да» сразу заводит новую запись с теми же параметрами.
- **ADMIN-промпты** — администратор пишет вручную или из шаблона:
  любой вопрос + до N вариантов ответа. Используется для
  опросов, напоминаний, акций.
- Шаблоны (`PromptTemplate`) — библиотека повторяющихся вопросов
- Ответы клиента приходят в раздел «Сообщения» и в Telegram
  администратору; счётчик непрочитанных в навигации
- Защита от спама: только один активный SYSTEM-промпт на клиента
  одновременно; повторный SYSTEM-промпт на ту же запись никогда
  не создаётся

### Склонение русских имён в обращениях

«Повторить запись **к Анне**» вместо «к Анна». Используется `petrovich`
с собственным детектором пола поверх библиотеки:

- Распознаёт пол по фамилии (`-ов / -ев / -ин / -ский` → м.;
  `-ова / -ева / -ская` → ж.)
- Список мужских имён на гласную (Илья, Никита, Лука, Саша, Дима…),
  которые petrovich иначе классифицирует как женские
- Поддержка трёхсложных «Имя Отчество Фамилия» по русской конвенции
- Все падежи: дательный для «к мастеру», другие — по месту

### Telegram-бот для админов и мастеров

Отдельный TypeScript-сервис на `grammy`, общая Prisma-схема с веб-частью.
Не дублирует админку, а даёт быстрый мобильный доступ.

- **Привязка по Telegram ID**: `Admin.telegramId` и `Master.telegramId` —
  бот идентифицирует пользователя по chat id, без логина
- **Инвайт-токены** — администратор генерирует одноразовую ссылку, мастер
  переходит по ней в боте и его аккаунт автоматически линкуется
- **Меню админа** — статистика на сегодня, ближайшие записи, переход
  по дням вперёд с инлайн-клавиатурами
- **Меню мастера** — только свои записи, свои действия (подтвердить /
  отменить / отметить выполненной)
- **Inline-управление**: смена статуса записи прямо в чате через
  callback-кнопки, без перехода в веб-админку
- Поллер опрашивает БД на новые записи каждые 30 секунд и шлёт пуш-нотификации

### Безопасность

- JWT в HttpOnly-cookies для админа и для клиентского кабинета —
  два разных секрета, два разных cookie
- `getJwtSecret()` ленивый, не валится на этапе `next build` без env
- `bcrypt` для паролей админов
- На HTTP-тестовом стенде — env-флаг `INSECURE_COOKIES=1` снимает
  Secure-флаг с cookie (без него браузер их не сохраняет по HTTP),
  на HTTPS-проде флаг убирается
- HTML-escape всех пользовательских данных в Telegram-сообщениях
- Каскадные удаления через Prisma `onDelete: Cascade` /  `SetNull`
  для безопасных удалений сущностей с историей

### Force-dynamic для каталогов

Серверные компоненты, читающие из Prisma (`/services`, `/shop`,
`/school`), помечены `export const dynamic = 'force-dynamic'` —
иначе Next пытается пререндерить их на этапе `npm run build`, когда
БД-контейнер недоступен; `try/catch` ловит ошибку, возвращает пустой
массив, и в статику запекается «нет данных». Урок учтён, директива
обязательна для любой новой страницы с Prisma.

## Инфраструктура и деплой

- VPS, Docker Compose, три сервиса:
  - `db` — `postgres:16-alpine`, только внутренняя сеть, том `pgdata`
  - `web` — Next.js приложение, через внешнюю сеть `ghuauld`
    проксируется отдельным nginx-контейнером на 80/443
  - `bot` — Telegram-бот, отдельный Dockerfile в `tg_bot_hair/`
- `docker-entrypoint.sh` на запуске контейнера: `prisma db push --accept-data-loss`
  → `npm start`. Схема применяется автоматически на каждый deploy
- `.dockerignore` обязателен: исключает `node_modules`, `.next`, `.git`,
  `.env*`. Без него `COPY . .` затирает свежеустановленные зависимости
  устаревшим host-`node_modules`
- Деплой одной серией команд:
  ```bash
  cd /var/www/hair-atelier
  git pull origin main
  docker compose up -d --build web
  docker compose logs -f web
  ```
- Pgdata в Docker-томе; `pg_dump`-бэкапы выносятся на хост по cron
